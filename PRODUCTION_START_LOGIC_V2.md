# ?? Зміна логіки СТАРТ виробництва (v2)

## ?? Що змінилося?

### ? Було (v1):
```
СТАРТ = Дата замовлення + 2 РОБОЧІ дні (фіксовано)
```

### ? Стало (v2):
```
СТАРТ = Дата замовлення + 1 РОБОЧИЙ день (мінімум)
    + ?накопичене_перевищення / потужність? днів (якщо є)
```

---

## ?? Нова логіка

### Крок 1: Базовий старт
```csharp
var baseStart = AddWorkingDays(orderDate, 1);
// Завжди мінімум +1 робочий день від замовлення
```

### Крок 2: Перевірка перевантаження
```csharp
double accumulatedOverflow = 0;

// Дивимося на всі дні ДО базового старту
foreach (var day in productionLoad.Where(x => x.Key < baseStart))
{
    var overflow = day.Value - capacity;
    if (overflow > 0)
        accumulatedOverflow += overflow;
}
```

### Крок 3: Розрахунок затримки
```csharp
if (accumulatedOverflow > 0)
{
    // Округлюємо вгору
    var delayDays = (int)Math.Ceiling(accumulatedOverflow / capacity);
    actualStart = AddWorkingDays(baseStart, delayDays);
}
else
{
    actualStart = baseStart; // Немає затримки
}
```

```
Розрахунок:
1. Базовий старт: 14.10 + 1 робочий день = 15.10 (середа)

4. СТАРТ: 15.10.2025 (середа)
```

### Приклад 2: Перевантаження 500 м?
```
Замовлення: 14.10.2025 (вівторок)
Попередні замовлення: перевищення 500 м?
Потужність: 1000 м?/день

Розрахунок:
1. Базовий старт: 14.10 + 1 робочий день = 15.10 (середа)
2. Накопичене перевищення: 500 м?
3. Затримка: ?500 / 1000? = ?0.5? = 1 день
4. СТАРТ: 15.10 + 1 робочий день = 16.10.2025 (четвер)
```

### Приклад 3: Велике перевантаження 2500 м?
```
Замовлення: 14.10.2025 (вівторок)
Попередні замовлення: перевищення 2500 м?
Потужність: 1000 м?/день

Розрахунок:
1. Базовий старт: 14.10 + 1 робочий день = 15.10 (середа)
2. Накопичене перевищення: 2500 м?
3. Затримка: ?2500 / 1000? = ?2.5? = 3 дні
4. СТАРТ: 15.10 + 3 робочі дні = 20.10.2025 (понеділок)
```

## ?? Як це працює на практиці?

### Сценарій: 3 замовлення підряд

#### Замовлення 1:
```
Замовлено: 14.10 (вт)
Площа: 1500 м?
Потужність: 1000 м?/день

СТАРТ: 14.10 + 1 = 15.10 (ср) ? немає попередніх
Розподіл:
  - 15.10: 1000 м? ? перевищення 0
  - 16.10: 500 м? ? перевищення 0
Накопичене перевищення: 0 м?
```

#### Замовлення 2:
```
Замовлено: 15.10 (ср)
Площа: 1200 м?
Потужність: 1000 м?/день

Базовий старт: 15.10 + 1 = 16.10 (чт)
Перевірка перевантаження до 16.10:
  - 15.10: 1000/1000 ? 0
  - 16.10: 500/1000 ? 0 (але це базовий старт, не рахуємо)
Накопичене: 0 м?
СТАРТ: 16.10 (чт) ? накладається на Замовлення 1

Розподіл:
  - 16.10: 500 (вже зайнято) + 500 = 1000 м? ? перевищення 0
  - 17.10: 700 м?
Накопичене перевищення: 0 м?
```

#### Замовлення 3:
```
Замовлено: 16.10 (чт)
Площа: 1000 м?
Потужність: 1000 м?/день

Базовий старт: 16.10 + 1 = 17.10 (пт)
Перевірка перевантаження до 17.10:
  - 15.10: 1000/1000 ? 0
  - 16.10: 1000/1000 ? 0
  - 17.10: 700/1000 ? 0 (це базовий старт)
Накопичене: 0 м?
СТАРТ: 17.10 (пт) ? накладається на Замовлення 2

Розподіл:
  - 17.10: 700 (вже зайнято) + 300 = 1000 м? ? перевищення 0
  - 18.10: 700 м?
```

## ? Переваги нової логіки

1. **Більш швидкий старт** - мінімум +1 день замість +2
2. **Автоматична компенсація** перевантаження попередніх замовлень
3. **Пропорційний розрахунок** затримки на основі реального перевищення
4. **Уникнення колізій** - замовлення не накладаються понад потужність

- [ ] Всі дати - робочі дні
